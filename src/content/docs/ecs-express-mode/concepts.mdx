## What Is Express Mode?

<Tooltip term="Express_Gateway_Service">ECS Express Mode</Tooltip> is a simplified deployment model that removes the need to provision separate ALB, target group, and listener resources. You create one resource — `aws_ecs_express_gateway_service` — and ECS provisions a managed <Tooltip term="Shared_ALB">Application Load Balancer</Tooltip>, configures health checks, sets up auto scaling, and assigns a unique <Tooltip term="Application_URL">Application URL</Tooltip> on `*.ecs.<region>.on.aws`.

In classic ECS, you wire together `aws_lb`, `aws_lb_target_group`, `aws_lb_listener`, `aws_ecs_service`, and `aws_appautoscaling_target`. Express Mode collapses all of this into one resource.

<Aside type="note">
  Express Mode is specifically designed for stateless HTTP/HTTPS web
  applications and APIs. It is not a general-purpose replacement for all ECS
  workloads.
</Aside>

The <Tooltip term="Shared_ALB">Shared ALB</Tooltip> hosts up to 25 Express services in the same <Tooltip term="VPC">VPC</Tooltip> using Host header routing. The first service in a VPC determines whether the ALB is internet-facing or internal.

## Supported Workloads

Express Mode works for:

- **REST / OpenAPI APIs** — stateless request/response over HTTPS
- **Stateless web applications** — frontends, dashboards, documentation sites
- **Rapid prototyping** — public HTTPS endpoint without infrastructure boilerplate
- **Developer self-service** — deploy HTTP services without platform team help

## Unsupported Workloads

Use classic ECS for these:

- **Batch / queue workers** — no HTTP endpoint, no ALB needed
- **gRPC** — ALB terminates HTTP/2, breaking gRPC streaming
- **Raw TCP/UDP** — Express Mode only handles HTTP/HTTPS
- **Non-HTTP services** — anything that cannot respond to ALB health checks

<Aside type="caution">
  Deploying a non-HTTP workload with Express Mode results in perpetual health
  check failures and rollback loops. Use classic ECS with an NLB or no load
  balancer.
</Aside>

## Single Subnet Limitation

Classic ECS lets you place the ALB in public subnets while running tasks in private subnets. <Tooltip term="Express_Gateway_Service">Express Mode</Tooltip> does not support this split — it uses one subnet list for both.

- **<Tooltip term="Public_Subnet">Public subnets</Tooltip>** (route to <Tooltip term="Internet_Gateway">IGW</Tooltip>) → internet-facing ALB, tasks with public IPs
- **Private subnets** (no IGW route) → internal ALB, reachable only within the <Tooltip term="VPC">VPC</Tooltip>

<Aside type="caution">
  If you need public ALB with private tasks, use classic ECS with separate
  subnet configurations.
</Aside>
