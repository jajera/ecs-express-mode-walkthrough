## Express vs Classic ECS

Use this checklist to determine whether <Tooltip term="Express_Gateway_Service">Express Gateway Service</Tooltip> or a classic ECS service with self-managed ALB is the right fit for your workload. The table below compares the two approaches across the dimensions that most often influence the decision.

<Aside type="tip">
  If your application is a stateless HTTPS web app or REST API and you do not
  need split public/private subnet placement, Express Mode eliminates
  significant infrastructure wiring.
</Aside>

## How to Use This Checklist

Review each dimension against your workload requirements:

1. **If all Express Mode traits match your needs** — use Express Mode for faster setup and less infrastructure code
2. **If any Classic ECS trait is required** — use a classic ECS service with self-managed ALB and networking
3. **If unsure about protocol requirements** — Express Mode only supports HTTP/HTTPS; any gRPC, TCP, or UDP traffic requires classic ECS

<Aside type="caution">
  The first{" "}
  <Tooltip term="Express_Gateway_Service">Express Gateway Service</Tooltip>{" "}
  deployed in a VPC determines whether the{" "}
  <Tooltip term="Shared_ALB">Shared ALB</Tooltip> is internet-facing or
  internal. All subsequent Express services in that VPC share the same ALB type.
</Aside>
